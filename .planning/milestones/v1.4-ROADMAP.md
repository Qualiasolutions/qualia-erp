# Milestone v1.4: Admin Portal Onboarding

**Status:** SHIPPED 2026-03-09
**Phases:** 17-19
**Total Plans:** 8

## Overview

Streamlined admin workflow to convert ERP projects to portal-accessible projects with automated client invitation and onboarding flow. Admin can import projects, configure portal settings, send branded invitation emails, and clients can create accounts with immediate project access.

## Phases

### Phase 17: Project Import Flow

**Goal**: Admin can configure ERP projects for portal access and prepare them for client invitation
**Depends on**: Phase 16
**Plans**: 3 plans

Plans:

- [x] 17-01: Admin import page with project list and portal status filtering
- [x] 17-02: Project selection and client roadmap preview modal
- [x] 17-03: Portal settings configuration and persistence with visual confirmation

**Details:**

- Server action `getProjectsForPortalImport()` with auth checks and FK normalization
- 3-tier badge system: Portal Active (green), Portal Ready (teal), Not Configured (gray)
- Bulk selection toolbar with Set-based state management
- RoadmapPreviewModal showing client-facing phase timeline
- PortalSettingsModal with welcome message, visibility toggles (roadmap/files/comments)
- Portal settings stored in project.metadata JSONB column
- Zod validation via portalSettingsSchema
- Activity logging for configuration changes

### Phase 18: Invitation System

**Goal**: Admin can invite clients and track invitation lifecycle
**Depends on**: Phase 17
**Plans**: 3 plans

Plans:

- [x] 18-01: Database schema and server actions for invitation tracking
- [x] 18-02: Invitation email template and admin UI for sending invitations
- [x] 18-03: Invitation status badges and history modal with resend capability

**Details:**

- client_invitations table with invitation_status enum (sent/resent/opened/accepted/expired)
- Secure token generation via crypto.randomUUID()
- Idempotent invitation creation (returns existing if already sent)
- UNIQUE(project_id, email) constraint prevents duplicates
- RLS policies: admin full access, managers project-specific view
- Branded HTML email template via Resend with Qualia teal gradient
- SendInvitationModal with email input, welcome message preview, visibility preview
- InvitationHistoryModal (376 lines) with timeline view and resend functionality
- 4-state color-coded status badges (blue/purple/amber/green)
- Status badge clicks open full history modal

### Phase 19: Client Onboarding Flow

**Goal**: Client can create account and access project immediately after invitation
**Depends on**: Phase 18
**Plans**: 2 plans

Plans:

- [x] 19-01: Signup page with invitation validation and account creation
- [x] 19-02: Auto-login verification and full portal access testing

**Details:**

- Server-side invitation token validation via getInvitationByToken()
- Branded two-panel signup page matching login aesthetic
- Pre-filled email (read-only), password validation (min 8 chars), confirm password
- Admin client for profile/client_projects creation (RLS bypass during signup)
- Atomic account creation: auth user -> profile (role='client') -> client_projects link
- Auto-login via signUp() session creation (no separate signIn needed)
- window.location.href redirect to /portal/[projectId] for cookie persistence
- Middleware updated to redirect authenticated users from /auth/signup
- Comprehensive verification document (476 lines) with manual testing checklist
- 6 components verified: middleware, auto-login, redirect, routing, access control, features

## Milestone Summary

**Key Decisions:**

- Store portal settings in project.metadata JSONB (flexible schema, no new table)
- Three-tier badge system (Active/Ready/Not Configured) for clear visual status
- Secure token generation with crypto.randomUUID() (built-in Node.js)
- Idempotent invitation creation (prevents duplicates)
- Admin client for profile creation during signup (RLS bypass)
- window.location.href for post-signup redirect (ensures cookie transfer)
- Role-based routing reads role from database (not JWT) for real-time enforcement

**Issues Resolved:**

- Middleware not redirecting authenticated users from /auth/signup (fixed in 5039391)
- TypeScript type errors from database type regeneration (fixed in 5039391)
- ESLint error with 'use client' directive mid-file (split into server/client components)
- Checkbox indeterminate state TypeScript error (fixed Radix pattern)

**Issues Deferred:**

- Token expiry for invitations (deferred to v2 — INVITE-09)
- Email confirmation after signup (disabled for friction-free onboarding)
- Rate limiting on signup attempts per token
- Manual testing of full signup flow (checklist provided)

**Technical Debt Incurred:**

- None critical — all items are v2 features or optional security hardening
