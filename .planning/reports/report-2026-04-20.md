## Non-obvious findings (worth remembering)

- **`'use cache'` + SWR is a trap.** Task 2 of Phase 13 originally swapped `usePortalDashboard`'s fetcher to the `getCached*` wrappers in `lib/cached-reads.ts`. Build failed: `cached-reads.ts` imports `createAdminClient` (server-only), and importing the module into `lib/swr.ts` (bundled client-side because `'use client'`) leaks server code into the browser bundle. The right split: server components use the cached wrappers on initial render; SWR client-side revalidation uses the plain server actions. Commit `0d5199c` documents the revert. Worth saving as a learning — anytime I want cached data in SWR, the cache lives in a separate server-only layer and SWR hits the server action that wraps the cache, not the cache directly.
- **Jest 29+ renamed `--testPathPattern` to `--testPathPatterns` (plural).** Several plan validation commands had the old singular form. Noted — worth fixing future plans.
- **Pre-deploy gate hook (`pre-deploy-gate.js`) can false-positive on `vercel --prod` substrings in heredocs.** A `cat > file.md <<'EOF'` that happened to contain the literal string `vercel --prod` in its content triggered the deploy-gate hook. Workaround: use `Write` tool or avoid the exact substring in shell content. Not a blocker, but confusing when it fires.
- **`pipeline.test.ts` mock pattern — when a server action adds a new shared import (like Phase 11 adding `getCachedUserRole` from `@/app/actions/shared`), the test's `jest.mock('@/app/actions/shared', ...)` must be updated to export the new symbol.** Missed this once; caught by failing tests and patched in commit `269f436`.
- **Post-deploy perf gains show mostly at authenticated pages.** Edge endpoints (home 307, login 200, health 200) improved 1-14% from indexes alone. The real 200-300ms-per-page wins from portal-cache adoption only matter for authenticated portal navigation. Baseline: 0.29s median home, 0.28s login, 0.71s health. Post-deploy: 0.25s, 0.27s, 0.70s. Real clients will feel it more.
- **Fawzi kept CRM clients + projects.** The Home "Workspaces" grid builds from CRM clients joined with projects.client_id — so even though 16 client logins are wiped, the admin view will still render cards for the 22 clients with projects. Flagged to Fawzi; no action taken (he wanted to keep both intact).
- **Task 2 cross-wave dependency danced cleanly.** Task 2 (Wave 1) created `getCachedProjectIntegrationStatus` in `lib/cached-reads.ts`; Task 1 (also Wave 1) imported it in `app/(portal)/projects/[id]/page.tsx`. Both parallel, both committed, imports resolved at build time not write time. Works because builders scoped to different files.
