  <action>
    Make three targeted changes to portal-admin-panel.tsx:

    **Change 1 — Confirmation guard on handleRemoveAccess:**
    At the top of `handleRemoveAccess`, before the optimistic state update, add:
    ```typescript
    const confirmed = window.confirm(
      `Remove ${removed?.client?.full_name || 'this client'}'s access to ${removed?.project?.name || 'this project'}? This cannot be undone.`
    );
    if (!confirmed) return;
    ```
    Use `assignments.find(a => a.id === assignmentId)` BEFORE the optimistic removal so `removed` is populated for the message.

    **Change 2 — Separate isPending flag for project creation (if 20-01 added isCreatingProject):**
    The invite form uses `isPending` from `useTransition`. The project creation form introduced in 20-01 uses a separate `isCreatingProject` state. Verify the invite button's `disabled` prop uses `isPending` and the create project button uses `isCreatingProject` — no cross-contamination where one spinner blocks the other form.

    If 20-01 used `startTransition` for the project creation (which shares `isPending`), refactor project creation to use a separate `useState<boolean>` flag for `isCreatingProject` instead, and call `setIsCreatingProject(true/false)` explicitly. This prevents the invite form going disabled while a project is being created.

    **Change 3 — Ensure all toast.error calls use `result.error` not a fallback-only string:**
    Audit every `toast.error()` call in the file. Pattern to check: `toast.error(result.error || 'Fallback')`. This is correct — if result.error is populated, it shows the specific DB message. If any call is `toast.error('Hardcoded message')` without using `result.error`, update it to `toast.error(result.error || 'Hardcoded message')`.

    Do NOT change the existing optimistic rollback logic for remove-access — it works correctly.
