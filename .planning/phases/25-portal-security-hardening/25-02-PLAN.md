**Fix 1 — `markInvitationOpened` and `markInvitationAccepted` use token not UUID PK (`app/actions/client-invitations.ts`):**

Read the full `client_invitations` table schema to confirm a `token` column exists (check by looking at existing `getInvitationByToken` query — it queries `.eq('invitation_token', token)` which confirms the column).

Change `markInvitationOpened(invitationId: string)`:

- Rename parameter to `token: string`
- Add UUID format guard at top: `if (!/^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$/i.test(token) && token.length < 20)` — actually, since tokens are opaque strings (not UUIDs), just add a basic sanity check: `if (!token || token.length < 10) return { success: false, error: 'Invalid token' };`
- Change the update query from `.eq('id', invitationId)` to `.eq('invitation_token', token)`

Change `markInvitationAccepted(invitationId: string)`:

- Rename parameter to `token: string`
- Same sanity check as above
- Change `.eq('id', invitationId)` to `.eq('invitation_token', token)`
- Add `.in('status', ['opened', 'sent', 'resent'])` filter to prevent re-accepting an already-accepted invitation

Make `markInvitationAccepted` internal-only: add a JSDoc comment `/** @internal — called only from signup flow, not exposed as a public API */`. Do NOT export it differently (it already uses 'use server' which is fine for server-only). The comment is sufficient.

**CRITICAL — Update callers to pass token instead of UUID PK:**

Both `markInvitationOpened` and `markInvitationAccepted` are called with `invitation.id` (UUID PK) today. After renaming the parameter to `token`, you MUST update these callers:

1. `app/auth/signup/page.tsx` (line ~58): calls `markInvitationOpened(invitation.id)`. The token is available as the URL search param (the `token` variable used to call `getInvitationByToken`). Change to `markInvitationOpened(token)` where `token` is the URL param.

2. `app/actions/auth.ts` (line ~202): calls `markInvitationAccepted(invitation.id)`. The token must be threaded through to this call. If the function receives `invitationToken` as a parameter, use it directly. Otherwise, update `getInvitationByToken` to also return the token in its result object (add `invitation_token` to the select), then pass `invitation.invitation_token` to `markInvitationAccepted`.

**Why this matters:** Both old and new parameters are `string` — TypeScript will NOT catch this. If callers still pass UUID PKs into token-based queries, the update silently matches zero rows. Invitation status will never update, breaking the invite flow.

Read both caller files before editing to understand the full call chain.

**Fix 7 — `sendClientPasswordReset` verify client role (`app/actions/client-portal.ts`, line ~400):**

After the `isUserManagerOrAbove` check passes, add a lookup to verify the target email belongs to a client-role profile:

```ts
const normalizedEmail = email.trim().toLowerCase();

// Verify target is a client profile before sending reset
const { data: targetProfile } = await supabase
  .from('profiles')
  .select('role')
  .eq('email', normalizedEmail)
  .single();

if (!targetProfile) {
  return { success: false, error: 'No portal account found for this email' };
}

if (targetProfile.role !== 'client') {
  return { success: false, error: 'Password reset is only available for client accounts' };
}
```

Then proceed with `supabase.auth.resetPasswordForEmail(normalizedEmail, ...)`.
