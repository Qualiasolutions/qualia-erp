<objective>
Eliminate the DB query for user role in middleware by storing the role in JWT custom claims via a Supabase auth hook.

Purpose: Every authenticated request currently makes a `profiles` table query just to get the user's role. This adds ~50-100ms latency to every page load. A Supabase custom access token hook injects the role into the JWT, making it available instantly.

Output: Migration file with the hook function + rewritten middleware that reads role from claims.
</objective>

  <name>Task 1: Create Supabase custom access token hook migration</name>
  <files>supabase/migrations/20260327000000_custom_claims_hook.sql</files>
  <action>
Create a PostgreSQL function `custom_access_token_hook` that Supabase calls when generating access tokens.

The function signature MUST be:

```sql
CREATE OR REPLACE FUNCTION public.custom_access_token_hook(event jsonb)
RETURNS jsonb
LANGUAGE plpgsql
STABLE
AS $$
```

Inside the function:

1. Extract user ID from `event->'user_id'` (it's a TEXT, not UUID — cast appropriately)
2. Query `public.profiles` for the user's `role` column
3. If role found, set it in the claims: `event := jsonb_set(event, '{claims,user_role}', to_jsonb(role_value))`
4. If no role found (new user, deleted profile), set a default: `'employee'`
5. Return the modified event

Also grant execute to `supabase_auth_admin`:

```sql
GRANT USAGE ON SCHEMA public TO supabase_auth_admin;
GRANT EXECUTE ON FUNCTION public.custom_access_token_hook TO supabase_auth_admin;
GRANT SELECT ON TABLE public.profiles TO supabase_auth_admin;
```

And revoke from anon/authenticated to prevent direct calls:

```sql
REVOKE EXECUTE ON FUNCTION public.custom_access_token_hook FROM authenticated, anon, public;
```

Apply this migration via Supabase MCP or CLI.

IMPORTANT: After migration, the hook must be enabled in the Supabase Dashboard under Authentication -> Hooks. This is a `checkpoint:human-action` that the executor will handle via user_setup.
</action>

Rewrite the middleware to eliminate the `profiles` DB query for role.

Current flow (SLOW):

New flow (FAST):

2. Read role from `data?.claims?.user_role` (set by the custom hook)
