  <action>
framer-motion provides a `LazyMotion` component + `m` namespace that enables tree-shaking and lazy feature loading. This is the official way to reduce bundle size.

**Step 1: Create `lib/lazy-motion.tsx`**

```tsx
'use client';

import { LazyMotion } from 'framer-motion';
import type { ReactNode } from 'react';

// Async import = true lazy loading. framer-motion's domAnimation features
// are NOT included in the initial JS bundle — they load on demand.
const loadFeatures = () => import('framer-motion').then((mod) => mod.domAnimation);

export function LazyMotionProvider({ children }: { children: ReactNode }) {
  return (
    <LazyMotion features={loadFeatures} strict>
      {children}
    </LazyMotion>
  );
}

// Re-export m for convenience
export { m } from 'framer-motion';
export { AnimatePresence } from 'framer-motion';
```

CRITICAL: The `features` prop MUST be an async function (not a direct object import). When `features` is a function, LazyMotion defers loading until the first `m` component renders. When `features` is a direct object (like `domAnimation` imported at module scope), it loads synchronously and defeats the purpose.

- CORRECT: `features={loadFeatures}` where `loadFeatures = () => import('framer-motion').then(mod => mod.domAnimation)`
- WRONG: `features={domAnimation}` with `import { domAnimation } from 'framer-motion'` (synchronous, still in bundle)

We use `domAnimation` (not `domMax`) because the codebase only uses basic features (animate, exit, variants, gestures). `domAnimation` is ~17KB vs ~32KB for the full bundle.

**Step 2: Add LazyMotionProvider to the root layout**

In `app/layout.tsx`, import and wrap the children (or the body content) with `<LazyMotionProvider>`:

```tsx
import { LazyMotionProvider } from '@/lib/lazy-motion';
// ... in the JSX:
<LazyMotionProvider>{children}</LazyMotionProvider>;
```

Place it inside the body tag, wrapping the main content area. It should wrap everything that might contain animated components.
