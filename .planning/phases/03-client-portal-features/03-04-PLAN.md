  <action>
Create server actions for activity logging:

1. **getProjectActivityFeed(projectId: string, clientVisibleOnly = false, limit = 50)** → ActionResult<ActivityLog[]>
   - Query activity_log WHERE project_id = projectId
   - If clientVisibleOnly: add WHERE is_client_visible = true
   - Join with profiles for actor_id user info
   - ORDER BY created_at DESC (newest first)
   - LIMIT to specified limit (default 50)
   - Use normalizeFKResponse for profile join
   - Return array of activity entries with:
     - action_type (e.g., 'phase_completed', 'file_uploaded', 'comment_added')
     - action_data (JSON with event details)
     - actor (user profile)
     - created_at
     - is_client_visible

2. **createActivityLogEntry(data: { projectId: string; actionType: string; actionData?: any; isClientVisible: boolean; actorId?: string })** → ActionResult
   - Get current user ID from session (use as actorId if not provided)
   - Insert into activity_log table:
     - project_id
     - action_type
     - action_data (JSON)
     - is_client_visible
     - actor_id
     - created_at (auto)
   - revalidatePath(`/portal/${projectId}/updates`)
   - Return success

3. **Helper: formatActivityMessage(entry: ActivityLog): string**
   - Based on action_type, generate human-readable message:
     - 'phase_completed': "Phase {phase_name} completed"
     - 'phase_started': "Phase {phase_name} started"
     - 'file_uploaded': "File {file_name} uploaded"
     - 'comment_added': "Comment added on {phase_name}"
     - 'project_created': "Project created"
     - 'client_invited': "Client access granted"
   - Use action_data JSON to fill placeholders
   - Return formatted string

Follow existing action patterns with ActionResult, error handling, and revalidation.

Note: activity_log table already exists with columns: id, project_id, action_type, actor_id, action_data, is_client_visible, created_at.
</action>
