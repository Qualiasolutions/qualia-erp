  <action>
    1. Install: `npm install @sentry/nextjs`

    2. Create `sentry.client.config.ts` (project root):
       - `Sentry.init()` with `dsn` from `process.env.NEXT_PUBLIC_SENTRY_DSN`
       - `tracesSampleRate: 0.1` (10% of transactions — low volume internal app, keep costs down)
       - `replaysSessionSampleRate: 0` (no session replay needed for internal tool)
       - `replaysOnErrorSampleRate: 1.0` (replay on errors for debugging)
       - `integrations`: add `Sentry.replayIntegration()` for error replays only
       - Guard: only init if DSN is defined (so dev works without it)

    3. Create `sentry.server.config.ts` (project root):
       - `Sentry.init()` with `dsn` from `process.env.SENTRY_DSN`
       - `tracesSampleRate: 0.1`
       - Guard: only init if DSN is defined

    4. Create `sentry.edge.config.ts` (project root):
       - `Sentry.init()` with `dsn` from `process.env.SENTRY_DSN`
       - `tracesSampleRate: 0.1`
       - Guard: only init if DSN is defined

    5. Add to `.env.example`:
       ```
       # Sentry (Observability)
       NEXT_PUBLIC_SENTRY_DSN=
       SENTRY_DSN=
       SENTRY_AUTH_TOKEN=
       SENTRY_ORG=
       SENTRY_PROJECT=
       ```

    Note: Use `NEXT_PUBLIC_SENTRY_DSN` for client (browser needs it) and `SENTRY_DSN` for server/edge (same value, but server-only env var). DSN is safe to expose publicly per Sentry docs.

  </action>
  <verify>

    - `grep "Sentry.init" sentry.client.config.ts sentry.server.config.ts sentry.edge.config.ts` — all three have init calls
    - `grep "SENTRY_DSN" .env.example` — env vars documented
  </verify>
  <done>Three Sentry config files exist at project root with guarded init (no-op without DSN). Env vars documented in .env.example.</done>

  <files>

  <action>
    1. Update `next.config.ts`:
       - Import `{ withSentryConfig }` from `@sentry/nextjs`
       - Keep existing `withBundleAnalyzer` wrapper
       - Wrap the final export: `export default withSentryConfig(withBundleAnalyzer(nextConfig), sentryOptions)`
       - Sentry build options:
         - `org: process.env.SENTRY_ORG`
         - `project: process.env.SENTRY_PROJECT`
         - `silent: !process.env.CI` (quiet locally, verbose in CI)
         - `widenClientFileUpload: true` (upload more source maps)
         - `disableLogger: true` (tree-shake Sentry logger in production)
         - `automaticVercelMonitors: true` (auto-instrument Vercel crons)
         - `tunnelRoute: "/monitoring"` (bypass ad blockers by proxying Sentry requests through our domain)
       - Add `"@sentry/nextjs"` to `experimental.optimizePackageImports` array
       - Update CSP connect-src to allow `https://*.ingest.sentry.io` (or the tunnel handles it, but add as fallback)

    2. Create `app/global-error.tsx`:
       - This is Next.js App Router's root error boundary (catches errors in root layout)
       - Must be a client component (`'use client'`)
       - Import `* as Sentry` from `@sentry/nextjs`
       - `useEffect` calling `Sentry.captureException(error)` when error prop changes
       - Render a minimal full-page error UI:
         - Centered card with "Something went wrong" heading
         - "Try again" button calling `reset()`
         - "Go home" link to `/`
         - Use inline styles (not Tailwind) since global-error replaces the entire HTML shell
       - Export `{ default }` and also export the component wrapped in `Sentry.withErrorBoundary` is NOT needed — global-error.tsx IS the boundary itself, just captureException manually

    3. Run `npx tsc --noEmit` to verify no type errors.

    4. Run `npm run build` to verify Sentry plugin integrates without breaking the build (it will warn about missing SENTRY_AUTH_TOKEN — that's expected without env vars set).
