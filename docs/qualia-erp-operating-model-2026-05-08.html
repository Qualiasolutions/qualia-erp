Qualia ERP · operating model

Project work, shift reports, and clear responsibility.

The ERP is being reframed away from employee task-chasing and toward a cleaner system: employees clock in for fixed work hours, report what happened before clocking out, and admins judge progress through projects, milestones, reports, deadlines, and framework quality.

What changed

Tasks are not the main ERP model.

Internal framework tasks can still exist for agent execution, but employees and admins should think in projects, milestones, work updates, and shift reports.

Clock-out means reporting, not finishing.

The employee completes required daily hours. Before clocking out, they submit what happened: progress, blockers, investigation, unfinished work, and next handoff point.

Deadlines stay simple.

Track project deadlines, milestone dates, and employee submission dates. Avoid turning every phase/task into management overhead.

Daily routine

Fixed hours stay fixed. The ERP does not need to pretend every work session ends with finished software. The important truth is what happened during the shift.

Clock in

Select one or more projects and write the intended outcome for the shift.

Build with Qualia Framework

Use commands like /qualia-plan, /qualia-build, /qualia-verify, and /qualia-report. The employee does not need to manually manage auth/RLS/backend terminology.

Submit report before clock-out

Every project touched in the shift needs a /qualia-report. The report can say pass, fail, blocked, partial, shipped, or investigated.

Clock out

Clock-out records time, submission date, and shift summary. This gives admin a clean record without requiring manual revision every time.

What reports mean

Progress

What changed in the repo/product, which milestone it supports, and whether the work is ready, partial, or blocked.

Verification

Build/test/deploy checks, /qualia-verify result, failures, and exact next step when a phase is not ready.

Submission date

The ERP can show who reported, when they reported, and whether clock-out was backed by real framework evidence.

How weakness becomes obvious

Do not rank people by raw task count.

That rewards shallow output. Use framework progress, verification failures, gap cycles, report quality, project difficulty, prompt discipline, and client-safe handoff quality.

Framework mastery

Which commands they can run correctly without admin rescue.

Prompting quality

Whether their instructions give agents clear context, scope, checks, and stop rules.

Verification quality

Whether they catch real broken behavior before handing work to admin or client.

Handoff quality

Whether another employee can continue from their report without rediscovering everything.

Rollout plan

Use the simplified menu

Employees stop hunting through admin/task pages. Their core flow is Dashboard, Projects, Knowledge.

Require reports for clock-out

No report means no complete shift record. The report is required even if the work is blocked.

Review weekly by project and person

Admin reviews project deadline risk, milestone risk, submitted reports, gap cycles, verification, and handoff quality.

Train against weak spots

Assign work based on proven framework mastery, not guesses. Weakness should be visible to both employee and admin.
