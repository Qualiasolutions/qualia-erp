How we work — tighter in May

Non-negotiable rules

01

Run /qualia-report before every clock-out

This is not optional. The ERP only knows you worked if you report. No report = no session credit. It takes 30 seconds. There is no exception.

02

Feature branches only — never push to main

Branch name: feat/description, fix/description. Create a PR. If you push directly to main, you explain it. The second time is a serious conversation.

03

Four-check verification after every deploy

HTTP 200, auth flow, no console errors, API latency under 500ms. Run /qualia-verify. Do not tell Fawzi something is deployed until all four pass.

04

RLS on every database table you create

No exceptions. If you add a table without Row Level Security policies, the PR does not merge. Check: ALTER TABLE x ENABLE ROW LEVEL SECURITY.

05

Zod validation on every server action

No raw FormData or JSON.parse reaching the database. Every input goes through a Zod schema. Use parseFormData() from lib/validation.ts.

How things should go

06

Read before you write — always

Open the project's CLAUDE.md and .planning/STATE.md before touching a single file. If the project has a DESIGN.md, read it before any frontend change.

07

30-minute stuck rule

Try /qualia-idk first — it often unblocks you. If you're still stuck 30 minutes later, message Fawzi. Do not spin for 2 hours in silence. That's wasted time for everyone.

08

MVP first, no gold-plating

Build only what was asked. Don't add features, refactor nearby code, or "while I'm at it" anything. A bug fix is a bug fix. A feature is exactly the feature specified.

09

Update PROGRESS.md at start of day, not end

What you plan to do today goes in at 9am. What you actually did gets checked off as you go. If Fawzi checks your PROGRESS.md at noon, it should reflect your morning.

10

Qualia is a company — never a "studio"

External copy, proposals, expo materials, any client-facing text: say "Qualia Solutions" or "company." Never studio, never agency, never shop. This matters for positioning.

How to report

Clock-out protocol
The command
# Navigate to the project you worked on
cd ~/Projects/sakani     # or maud, GlluzTech, etc.

# Run the report — this is mandatory at clock-out
/qualia-report

# The framework POSTs a structured session report
# to portal.qualiasolutions.net/api/v1/reports
# Fawzi sees it in the ERP under "Framework Reports"

Run in the project directory — not from ~/Projects

API key is read automatically from ~/.claude/.erp-api-key

Each session gets a QS-REPORT-NN ID for tracking

Retries are safe — idempotency key prevents duplicates

What the report captures

Project name and framework project ID

Git commits made during the session

Phase status changes (planned → executed → verified)

Milestones opened or closed

Blockers and gap cycles

Session duration and activity summary

# Check your framework setup is configured:
cat ~/.claude/.qualia-config.json

# Update framework if needed:
npx qualia-framework-v2 update
npx qualia-framework-v2 version
What Fawzi sees

Every report appears in the ERP under Admin → Framework Reports

Each row shows: project, report ID, phase, milestone, session summary

Missing reports are visible — if you clocked in but no report arrived, it shows as a gap

Dry-run reports (for testing) are filtered from production view automatically

Reports feed into client billing and project status tracking

During the day — quick commands
# When you start work on a project:
/qualia        # tells you exactly what's next

# For a task you can do in one session:
/qualia-quick  # fast all-purpose workflow

# If you don't know what to do:
/qualia-idk    # framework diagnoses your situation

# If something is broken:
/qualia-debug  # structured diagnosis

# End of session — non-negotiable:
/qualia-report
