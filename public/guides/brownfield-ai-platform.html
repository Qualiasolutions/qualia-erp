Existing Platform

Modify Existing AI Platform

Add features to multi-user platform (be careful!)

! WARNING: Platform Changes Affect Everyone

Platforms have multiple users. A bug affects ALL of them. Always test thoroughly and deploy during low-traffic times.

1 Go to the Project Folder
cd ~/Desktop/Projects/platforms/existing-platform
2 Check Git Status
git status && git branch
3 Start Claude Code
claude
4 Explore the Codebase Thoroughly
/brownfield-explore
"I need to add a usage analytics dashboard. Show me how the platform is structured, how data is isolated per organization, and where analytics might fit."

For deeper understanding:

/gsd:analyze-codebase
5 Create Feature Branch
git checkout -b feature/usage-analytics
6a For Small Changes (UI tweaks, admin features)
/gsd:quick
"Add a usage chart to the admin dashboard showing messages per day for the last 30 days. Only super admins should see this."
6b For Medium Changes (new features)
/gsd:new-milestone
"Add usage analytics feature. Each organization should see their own usage stats. Include: messages sent, API calls made, and cost estimate."

Then work through phases:

/gsd:discuss-phase 1 /gsd:plan-phase 1 /gsd:execute-phase 1
6c For Big Changes (database, auth, billing)

Do extra planning and run production audit first:

/pr

Then use full GSD with careful testing.

7 If You Need Database Changes
/sb
"Add a usage_logs table to track API calls per organization. Include: org_id, endpoint, tokens_used, cost, created_at. Add RLS so each org only sees their own data."
8 Test As Different Users

Test your changes as:

  • Super admin
  • Organization admin
  • Regular user
  • User from DIFFERENT organization (check isolation!)
9 Verify Data Isolation

Ask Claude to check:

"Check if User A from Organization 1 can see any data from Organization 2. Test the new usage_logs table specifically."
10 Deploy to Staging First
/git-flow

Deploy to staging and test there before production.

11 Deploy to Production (Off-Peak Hours)
/full-deploy

Monitor for 24 hours. Be ready to rollback.

Platform Change Checklist

  • Explored codebase first
  • Understood multi-tenant structure
  • RLS policies correct
  • Tested as all user types
  • Data isolation verified
  • Deployed to staging first
  • Production deploy off-peak
  • Monitoring active
